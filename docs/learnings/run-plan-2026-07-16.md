# Unified Run Plan — 7,758 OK Confederate Pensioners

**Run date:** 2026-07-16 (planned)
**Operator:** Jeremy Morris
**Mode:** unattended, headless=False, throttle=1.0s

## Goal

For every OK CW pensioner in `docs/research/digitalprairie/ok_pensioners.json`
(7,758 records), search Find a Grave and the Confederate Graves Registry.
Produce a final report + outliers.jsonl that we don't need to re-run.

## Per-pensioner flow

```
For each pensioner:
  ├─ load CGR blocking index (built once, 2,593 vets)
  ├─ lookup in blocking (instant; ~13ms / pensioner)
  ├─ build ranked_candidates list from CGR with match_strength annotations
  ├─ run 12 FaG strategies (sequential, throttled 1.0s between)
  │  ├─ B1-exact, B2-middle-initial, B3-first-initial-fuzzy
  │  ├─ B4-fuzzy-last, B5-apostrophe, C1-cw-context
  │  └─ F1a-birthyear, F1b-deathyear, F1c-year-sniper, F1d-year-window
  │     F2-regiment-bio, F3-nickname
  ├─ pick top 20 candidates, score each, sort by score
  ├─ decide fag_status (auto_accept / ambiguous / too_many / no_results / error)
  ├─ compute BOTH MATCH against CGR data (corroboration or direct_link)
  ├─ flush state.jsonl line (per pensioner, ~ms)
  ├─ flush outliers.jsonl line if outlier (low_score or no_results)
  └─ emit heartbeat every 50 pensioners (rate, ETA, file size)
```

## Resume safety

- `state.jsonl` is flushed **per pensioner** (atomic within thread).
- If the run is interrupted (PowerShell Ctrl-C, PC sleep, browser
  crash, anything), restart with the same `--out` and we resume
  from the last unflushed pensioner (which has been flushed).
- The first thing the new run does is `ResumeTracker` reads
  `state.jsonl` and skips completed pensioners.

## Failure modes and mitigations

| Failure | Mitigation |
|---|---|
| Cloudflare challenge (CAPTCHA) | `search_fag.py` waits 30s + retries. If still blocked, marks pensioner `error`. Outliers file gets it. |
| Network drop | Per-pensioner catch + status=error → outlier |
| Browser crash mid-run | Resume skips done; new run opens new browser |
| Power loss / PC sleep | Resume picks up exactly where we left off |
| Outlier too_many | Same — human review via view.html |

## What we expect NOT to do

- We do **not** delete or modify `data/results/run_<timestamp>/`
  files after the run. Whatever's there is what's there.
- We do **not** re-run a "fix" pass. If the run was bad, we
  document why and start a NEW run in a new output dir.

## Post-run enrichment (planned)

After the main run, I'll write a follow-up script that:
1. Loads `state.jsonl` for the run
2. Loads `dixiedata_export.csv` (local human-verified CW data)
3. For each pensioner in the run, check if local DD has a memorial_id
4. Append `dd_in_local: true/false` to each state record
5. View.html already has the field — humans can filter to "not in DD"
   and concentrate on those
6. Write `dd_marked.jsonl` for export

This is the **lower bound on what we gain from this whole project** —
DD has the human-verified set, anything not in DD is our "new finds".
