## 1. The 5-layer impossible-date filter (J13) — defense in depth

The strategy ladder scored candidates by name + burial state. A
modern person with the same surname (alive in 2020, burial in
Florida) could score the same as an ACW veteran (burial in OK)
when both names match. The date filter adds FIVE independent
layers that each kill the modern-candidate noise:

  1. **URL filter**: `birthyear=1810&birthyearfilter=after&deathyear=1955&deathyearfilter=before`
     attached to the search URL — FaG's own filter rejects
     impossible-date candidates before they ever come back
  2. **Score zero**: candidates with impossible birth/death
     years score 0.0; they don't win the rank-1 spot
  3. **Parse-time drop**: the FaG-result parser drops the
     candidate entirely if the year is implausible (avoids
     carrying dead weight in `ranked_candidates`)
  4. **Reviewer UI badge**: meta-row shows "unknown (ACW window:
     birth 1810-1880, death 1861-1955)" so the reviewer knows
     when scoring is date-unanchored
  5. **Stats counter**: the report counts how many records have
     no birth/death years (0% in our data, but future-proof)

**Lesson:** defense-in-depth beats single-layer filters. Each
layer fails differently (URL filter can be bypassed by FaG
changing their API; score-zero can be wrong if the pensioner
record has bad dates; etc). Having all five means even if one
breaks the others catch it.

## 2. ACW date window — back the constants with real data

The original constants (`1820-1870` birth, `1861-1950` death) were
guesses. The **J13-research** slice replaced them with research-
backed values (`1810-1880` birth, `1861-1955` death) using:

  - 577 local ground-truth pairs (pensioner → confirmed FaG)
  - 1,135-row age-at-death distribution from `local_soldiers_with_fag.csv`
  - The full distribution showed birth peaks in 1840s, death peaks
    in 1910s-1920s; 95% within the new window

**Lesson:** every "magic constant" in the codebase should be
backed by real data in `docs/research/`. Speculative ranges
silently drop matches; research-backed ranges don't.

The constants live in `scripts/fag/filters.py::ACW_BIRTH_YEAR_MIN/MAX`
and `ACW_DEATH_YEAR_MIN/MAX`. Same triple (`1820/1870/1861/1950`)
had been duplicated in 2-3 places before — they disagreed by 2-7
years each. **Fix:** single source of truth in filters.py, all
other modules import.

## 3. Dixiedata is READ-ONLY — J14's central design decision

The original auto-enricher idea (run pipeline → look up DD match
→ WRITE the match back into ok_pensioners.json so future runs
start pre-populated) was **vetoed** because it mutates the
input. The input is a fixed historical record from the OK State
Archives. Anything that auto-writes to it is dangerous (corrupts
the source, makes runs non-reproducible, gets a different answer
on a re-run).

The post-pipeline pattern is strictly additive:

  - Input: `data/test/test-batch-*.json` (or a slice of
    ok_pensioners.json) is **never** touched.
  - Output: `results.jsonl` per record + `dd_match.json` sidecar
    for batch.
  - Reviewer: opens `view.html`, sees the DD badge inline.

This is the model that the spouse scrape (J15-S2) followed too:
read-only on FaG + DD + pension data; only ADD-SCRAPES records.

**Lesson:** "I want to enrich the input" usually means "I want to
add new data per record." The answer is a sidecar + a badge,
not a write-back to the input. Sidecars are reproducible
(clear them, rerun, same answer); writes to input are not.

## 4. View.html second-pass embed — the substring-in-comment bug

The most painful bug of the burst. The second-pass embed check
was:

```python
if 'id="embedded-results-jsonl"' not in text:
    # embed
```

This was a *naive substring check*. The source `view.html` has
the literal `id="embedded-results-jsonl"` inside JS comments (in
the docstring explaining how the embed works). So the check
returned `True` (substring present in comments) even when no
real `<script>` tag existed → second pass skipped embedding →
page rendered 0 cards.

This bug existed silently for several previous runs; it only
surfaced when es-fresh-run's 177-card view.html showed up empty
and I had to debug.

**Lesson:** detection of "is this thing embedded?" must require
the structural marker AND a content marker. Real embed blocks
look like `<script type="application/json" id="..."> { ... }`.
The regex `<script ... id="...">[^>]*>\s*\{` matches only real
embeds (comments never have both adjacent).

The same bug exists in THREE places in `run_unified.py` — the
J9 (results), J14 (dd_match), and J15-S2 (spouse_match) embeds
all had the same logic. Fixed all three. Added 5 regression
tests in `tests/test_embed_detection_bug.py`.

**Lesson 2:** when fixing a bug, **search the codebase for the
same pattern elsewhere**. One bug = three copies.

## 5. Subprocess-isolated playwright sessions

The spouse scrape subprocess (`scripts/cgr/spouse_compare.py`)
needs to spin a fresh `playwright.sync_api` browser — but the
per-pensioner search loop already opened and tore down a
sync_playwright session in the same Python process. Reusing
sync_playwright in the same process for the second step fails
with "Playwright Sync API inside the asyncio loop."

**Fix:** invoke the spouse scrape as a subprocess from
`scripts/pipeline/run_unified.py`. The subprocess gets a fresh
asyncio event loop, a fresh Python interpreter, and a clean
playwright instance. Cost: 1 extra Python startup (~300ms) per
batch — negligible vs the 41-minute FaG run.

**Lesson:** when you need TWO playwright sessions in the same
workflow, subprocess them. Don't try to share the asyncio
loop.

## 6. Top-N>1 catches name-collision matches (J15-S2 → #14)

Of 47 spouse matches in es-fresh-run, **3 came from rank > 1**:

  - **Belli Earls** (rank 2, strong) — top-1 was a same-name
    modern person, rank 2 was the right husband
  - **Sarah S. Ellis** (rank 2, strong) — same pattern
  - **Margaret C. Ellis** (rank 3, medium) — only the
    last name matched (captured "Tennessee Belle Maxwell Ellis";
    local was "Allison Ellis")

These would have been missed by the default top_n=1. The
reviewer sees the badge text `(mem 147909496, rank 2)` AND the
tooltip `(matched at rank 2; the top candidate was different)`
so they know to verify manually.

**Lesson:** name-collision is the rule, not the exception. ACW
veterans from 1860-1880 often share surnames with modern
people; the strategy ladder's name-match score is high for
both, but only the ACW one has a meaningful FaG page. Always
allow a wider net for spouse compare (top_n>=2) and surface
the rank in the UI.

## 7. Auto-relax as a quality multiplier (#15)

Of 177 E-batch pensioners, **88 (50%) triggered the auto-relax
check** (OK search returned no high-score candidate). Of those:

  - **28 (16%)** got widened to US (US returned more candidates
    than OK)
  - **16 (9%)** kept OK (US returned ≤ candidates)
  - 44 had no US search run (probably all the DD matches that
    don't need additional FaG candidates)

This is a huge quality multiplier. Before auto-relax, the OK-
only filter would have:
  - Missed any pensioner whose ACW burial was in another state
    (modern Oklahoma burials for ACW soldiers who died elsewhere)
  - Missed any pensioner where OK burial was sparse enough that
    the same-name modern noise dominated

After auto-relax, US acts as a fallback scope. The cost is
roughly 2x the FaG hits for the 50% that need it — manageable.

**Lesson:** hard filters (state_filter="OK") are wrong by default
for research code. They should be a starting point with
auto-broadening when the narrowed search fails to find good
candidates. Runners that don't auto-relax silently miss matches.

## 8. The 41-minute wall clock breakdown

For 177 pensioners at 1.5s throttle:

  - Browser init + warmup: ~5 sec
  - Per-pensioner search (~10 strategies × 1.5s): ~25 sec × 177
    = ~73 min IF fully sequential; but strategies share the
    browser session so the actual time is 41 min total
  - Auto-relax: +5 min (28 doubled-up searches)
  - CGR dedup: <30 sec (local sqlite lookup)
  - DD match: <30 sec (local sqlite lookup)
  - Spouse scrape (subprocess): ~110 sec for 132 pensioners
  - Second-pass view.html embed: <1 sec

The bottleneck is the per-pensioner search, dominated by FaG
navigation + parse. At 1.5s throttle × 10 strategies = 15s per
pensioner. The 41-minute wall clock is consistent with that
plus browser reset overhead every 250 records (didn't trigger
in this run; would have if the batch were longer).

**Lesson:** for the full 7,709 pensioner run, plan for ~30 hours
of wall clock at 1.5s throttle. With the auto-relax +
subprocess-isolated spouse scrape enabled. That's a multi-day
job; should probably split into overnight chunks.

## 9. Sidecar pattern for downstream comparisons

Every comparison step (DD, spouse, CGR) writes BOTH:

  - a per-record field on the JSONL record (`dd_match`,
    `spouse_match`, `cgr_dedup_status`)
  - a sidecar JSON file (`dd_match.json`, `spouse_match.json`,
    `cgr_fag_dedup.json`)

The per-record field is for the inline UI badge. The sidecar is
for batch analysis (which DD rows matched what, which spouse
captures came from which memorial pages).

The sidecar is **embedded as a JSON script block** in view.html
via the second-pass embed, so the page works under `file://`
without needing a fetch. The first-pass copy at the start of
the batch can't embed yet because the sidecars don't exist; the
second-pass embed at the end of the batch does the embed.

**Lesson:** always use the per-record + sidecar + embed pattern.
Don't embed the JSONL inline into the page (too big); DO embed
the sidecars (small enough to be inlined, lets the page work
without fetch).

## 10. The reviewer's flow

What the reviewer does now, end to end:

  1. Open `output/es-fresh-run/view.html` (works offline; file://)
  2. See a sortable list of 177 pensioners with status badges,
     DD badges, spouse-known badges, spouse-match badges
  3. Filter to `decided` to see the 6 auto-accepts, then
     `auto_accept + CGR dup` to see the strongest confirmed
     matches, then `spouse_matched` to see all 47 gold badges
  4. Click a pensioner to expand its candidates; each candidate
     shows FaG captured spouse + matching local spouse
  5. Click the gold `♥ Spouse match` badge tooltip to see the
     strength + rank + memorial ID
  6. Click "View source" button to see the full pension card
     JSON inline (no leaving the page)
  7. Click the FaG memorial ID to verify against the actual
     FaG page externally

That's the entire flow. No separate tools, no spreadsheets, no
manual cross-referencing. The reviewer can process ~50 pensioners
per hour when working through this list.

**Lesson:** the goal of every pipeline feature is to remove
manual steps from this flow. Each slice (J11-J15, #14, #15)
either added a step, removed a step, or made a step more
informative. Future features should be measured against this.

## Concrete next steps informed by these learnings

(Forward-looking notes for the next round.)

  - **Issue #13** (cache memorial responses) — the spouse scrape
    does 132 page hits for a 177-pensioner run; many pensioners
    reference the same husband (different widows → same man).
    Caching saves ~30-50% of those hits.
  - **Issue #16** (spouse-follow-up pane for deceased husbands) —
    187/7,709 pensioners are widow filings where the husband
    never drew a pension. The captured spouse memorial has his
    data; surfacing it as a separate review card (NOT a primary
    pensioner entry) helps review completeness without
    mislabeling.
  - **Larger batches** — at ~14 sec per pensioner × 7,709
    pensioners = ~30 hours. Should split into ~500-pensioner
    chunks, each taking ~2 hours, then merge the JSONL files
    for a single view.html.
  - **Auto-relax as default** — currently opt-in via
    `FAG_AUTO_RELAX=1`. Should become the default for the
    7,709-run; the legacy OK-only behavior can stay available
    via the env var being unset.
